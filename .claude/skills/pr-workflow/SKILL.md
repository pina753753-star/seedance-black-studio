# PR作成・反映の標準フロー

## 1. 実装前

- 対象ファイルの現状を確認してから変更する（推測で書き換えない）
- 変更範囲を必要最小限にする

## 2. 実装後のセルフチェック

- 構文チェック（`node --check` 等、対象言語に応じたもの）
- `git diff --check`
- `git diff --name-only` で、意図したファイルだけが変更されているか確認
- `git status --short`

## 3. レビュー

- 実装者と異なる側（Codex実装→Claude Codeレビュー、Claude Code実装→Codexレビュー）が理想。ただし軽微な変更はセルフレビューでも可
- 新規登録処理・DBトリガー・決済など影響範囲が大きい変更は、必ず異なる側のレビューを挟む
- レビューでは、変更が意図した範囲に収まっているか、既存の正常系を壊していないかを確認する

## 4. ユーザーへの提示

- 実際の差分（diff）を提示し、commit・pushの前に必ずユーザーの明示的な承認を得る
- 「commit・push可」で終わらせず、実際に承認の言葉を待つ

## 5. commit・push・マージ

- 承認後、わかりやすい日本語のcommitメッセージでcommit・push
- ブランチを作成し、PRを作成してmainへマージする
- マージ後、mergeコミットのハッシュを報告する

## 6. STATUS.mdへの記録

- 完了した作業を、STATUS.mdの既存の記載形式（見出し・PR番号・詳細）に合わせて追記する
- 未完了の関連タスクがあれば「残タスク」セクションにも追記する
- この追記もユーザーの承認を経てからcommit・pushする

## 7. 完了報告

- 変更内容の要約
- この変更が既存の正常系のどの処理パスに影響しうるか
- 次に確認すべきこと（実機確認が必要な点など）
