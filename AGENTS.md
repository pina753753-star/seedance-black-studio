# FlowVid Studio — Codex作業ルール

このリポジトリでCodexが作業する場合、作業開始前に必ず本ファイルを読むこと。
本ファイルは `CLAUDE.md`(Claude Code向けの作業ルール)の内容をCodex向けに書き写したものであり、両エージェントで同じ制約を守ることを目的とする。

## プロジェクトの技術構成

- フロントエンド: フレームワークなしの静的HTML/JS(`generate-prod.html` など)
- バックエンド: Vercel Serverless Functions(`api/` 配下)
- DB/認証/ストレージ: Supabase(Postgres + Auth + Storage)
- 決済: Stripe
- 動画生成API: OpenRouter(Seedance)
- 透かし(watermark)処理: Railway上の別サービス(`watermark-server/`, Node/Express/ffmpeg)。無料プランのみ透かしを付与し、有料プランは透かしなしの動画を利用する

**生成フローの中心的なファイル**は以下であり、変更時は特に慎重に扱うこと。

- `api/seedance-status.js`(生成タスクのポーリング・完了判定・課金確定・返金・watermark連携を担う中心ファイル)
- `generate-prod.html`(生成ページのUI・ポーリング表示・失敗時表示)
- `api/_lib/` 配下

## 必ず守ること

- Flow本体のコードは、明示的な指示がない限り触らない
- `package.json`、API route、Supabase関連、OpenRouter関連、README、`.env` / `.env.example` は、明示的な指示がない限り触らない
- **git add / commit / push / マージは、ユーザーの明示的な指示があるまで行わない。** 自動フック等をトリガーとして自律的に実行しないこと。指示を待つこと。
- 新規生成テストは行わない(実際の課金・動画生成を伴うテストはしない)
- 推測で断定せず、確認できないことは「確認できません」と書く
- 修正前に「原因・修正箇所・確認方法」を出す
- 修正後に「変更内容・確認方法・次の確認箇所」を出す

## 正常系を壊さないためのルール(恒久ルール)

2026年7月、複数回の修正で正常系(fal.ai廃止後の画像生成、UI表示など)を壊す事象が発生した。
個々の修正の中身が正しくても、全体として動き続けることの確認を怠ると簡単に壊れる。
そのため、以下を今後すべての変更作業で守ること。

1. 変更の完了報告には、「この変更が既存の正常系のどの処理パスに影響しうるか」を必ずコードレビューとして示す。関数がどう呼ばれて完了まで到達するか、処理の流れを具体的に追って説明すること。
2. 可能な限り、mainへのマージ前にVercelのpreviewデプロイで実際の動作確認を挟む。
3. 一つの修正が終わった直後に次の修正へ進まない。必ず「動くことの確認」を一区切り終えてから次に進む。
4. 特に `generate-prod.html` や `api/_lib/` 配下など、生成フローの中心的なファイルに触る際は、影響範囲をより慎重に扱う。
5. 新しい作業に着手する前に、その作業に関連する既存のコード・マイグレーションファイルを実際に開いて読み、現状の実装・正常系の挙動を把握してから着手する。推測で語らず、確認できないことは「確認できません」と書く。
