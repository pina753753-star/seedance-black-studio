# 05. 自動運用・設計チェックリスト

> **読むタイミング：** 新しい自動運用（スケジュール実行・自動生成・自動通知・自動リカバリなど）を設計するときだけ。

新しい自動運用（スケジュール実行・自動生成・自動通知・自動リカバリなど）を
**設計する段階**で使うチェックリスト。実装前にすべての項目へ答えを出すこと。
答えが出せない項目は「確認できません／未設計」と明記し、実装に進まない。

## 1. 重複防止の設計（最優先）

- [ ] 冪等性：同じ処理が2回実行されても結果が1回分になる設計か（冪等キー・一意制約）
- [ ] **重複生成** を防ぐ確認：生成リクエストにIDを付け、処理済みIDをスキップするか
- [ ] **重複課金** を防ぐ確認：課金は「成功を確認してから1回だけ」になっているか
- [ ] 再起動・再デプロイ時に、処理中だったジョブが二重実行されないか

## 2. 失敗時の設計

- [ ] リトライ上限（数字で決める。例：3回）
- [ ] リトライ間隔（数字で決める。例：指数バックオフ 2s/4s/8s）
- [ ] 上限到達後の挙動（停止して通知か、スキップして次へか）
- [ ] 失敗の記録先（ログ・DB・通知）

## 3. 停止・介入の設計

- [ ] 手動で止める手段（キルスイッチ・フラグ・環境変数）があるか
- [ ] 異常検知時に自動停止する条件があるか（連続失敗N回で停止など）
- [ ] 人間が確認してから進む「手動介入ポイント」をどこに置くか

## 4. コスト・回数の上限

- [ ] 1実行あたりの想定コスト（数字）
- [ ] 1時間・1日あたりの実行回数上限（数字）
- [ ] 上限超過時の挙動（停止・通知）

## 5. ログ・通知の設計

- [ ] 各実行の開始・成功・失敗が後から追える記録があるか
- [ ] 失敗時に誰に・どうやって通知されるか
- [ ] ログにAPIキー・個人情報を出していないか

## 6. 設計レビューの出し方

- **結論 → 理由 → 数字**（リトライ上限・コスト上限・実行回数など具体値）の順で提示する。
- 推測で断定しない。既存コードの挙動が不明な場合は「確認できません」と書く。
- 実装に進む前に：**原因（なぜ必要か）・修正箇所・確認方法**＋触るファイル／触らないファイルを提示し、承認を得る。
- 実装後に：**変更内容・確認方法・次の確認箇所** を報告する。

## 7. 禁止事項

- 承認前の実装着手
- git add / commit / push（指示があるまで）
- 新規生成テスト（課金が発生するため。検証はドライラン・コードリーディング・ログで行う）
- ただし、課金の発生しない動作確認（ビルドが通るか、対象ページがエラーなく表示されるか）は、実装後に必ず行う
- Flow本体・package.json・API route・Supabase・OpenRouter関連コードへの無断変更
