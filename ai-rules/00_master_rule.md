# 00. マスタールール（すべての作業の入口）

FlowVid Studio の開発・監査・ドキュメント整備を、どのAIモデル（Fable / Sonnet / その他）でも同じ品質で進めるための最上位ルール。
**すべての作業セッションの最初に、このファイルを読むこと。**

## 1. ルールファイルの参照順

| 順番 | ファイル | 使うタイミング |
|---|---|---|
| 1 | `00_master_rule.md`（本ファイル） | セッション開始時に毎回 |
| 2 | `01_before_edit_check.md` | ファイルを編集する直前に毎回 |
| 3 | `02_auto_flow_risk_check.md` | 自動化フローに関わる変更・設計時 |
| 4 | `03_code_audit_checklist.md` | 監査・リリース前・テンプレート販売前 |
| 5 | `04_error_report_format.md` | エラー・不具合を報告するとき毎回 |
| 6 | `05_auto_operation_design_checklist.md` | 新しい自動運用を設計するとき |

## 2. 回答形式の絶対ルール

- 回答は必ず **結論 → 理由 → 数字（件数・行数・具体値）** の順で書く。
- **推測で断定しない。** 推測を書く場合は「推測ですが」と明示する。
- 確認できないことは、ごまかさず **「確認できません」** と書く。

## 3. 修正の前後で必ず出すもの

- **修正前：** ①原因 ②修正箇所（ファイル名・行） ③確認方法 を提示し、承認を得てから修正する。
- **修正後：** ①変更内容 ②確認方法 ③次の確認箇所 を報告する。
- 修正の提案時は、**触るファイル一覧** と **触らないファイル一覧** を必ず明記する。

## 4. 触ってはいけない領域（明示的な指示がない限り）

- Flow本体のコード（生成フロー・画面ロジック）
- `package.json`
- API route（`api/` 配下、`app/` 配下のroute）
- Supabase 関連コード（`supabase/` 配下、`auth-config.js` などの接続設定）
- OpenRouter 関連コード（APIキー・呼び出し処理）
- `.env` / `.env.example`

## 5. Git・テストの禁止事項

- **git add / commit / push は、ユーザーの明示的な指示があるまで絶対に行わない。**
- `git status` / `git diff` / `git log` などの読み取り確認は可。
- **新規生成テスト（実際に動画・画像を生成するテスト）は行わない。** 課金が発生するため。

## 6. 自動化での重複防止（最重要の安全項目）

自動化に関わる変更では、以下を毎回確認する：

- [ ] 同じリクエストが2回送られたとき、**重複生成** が起きないか（冪等キー・重複チェックの有無）
- [ ] リトライ処理が **重複課金** を引き起こさないか
- [ ] 失敗時に無限ループ・無限リトライにならないか（リトライ上限の有無）
- [ ] 確認できない場合は「確認できません」と報告し、実行しない

## 7. このルールの位置づけ

- 本ルールと個別ルールが矛盾した場合、**ユーザーの直近の指示 > 本ファイル > 個別ルール** の順で優先する。
- 販売用テンプレートに同梱する際も、このファイルを起点として構成する。
