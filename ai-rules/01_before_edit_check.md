# 01. 編集前チェック（ファイルを触る直前に毎回）

> **読むタイミング：** コード・設定ファイルを編集する直前。読むだけ・調査だけの作業では不要。

どのファイルを編集する場合も、編集の**直前**にこのチェックを通すこと。
1つでも確認できない項目があれば、編集せずに「確認できません」と報告する。

## 1. 禁止領域チェック

- [ ] 対象は Flow本体のコードではないか
- [ ] 対象は `package.json` ではないか
- [ ] 対象は API route（`api/` 配下、`app/` のroute）ではないか
- [ ] 対象は Supabase 関連コード（`supabase/` 配下・接続設定）ではないか
- [ ] 対象は OpenRouter 関連コードではないか
- [ ] 対象は `.env` / `.env.example` ではないか
- [ ] 上記に該当する場合、ユーザーの明示的な指示があるか

## 2. 影響範囲チェック

- [ ] このファイルを読み込んでいる他のファイルを特定したか（grep等で確認）
- [ ] 変更範囲は依頼に対して最小か（依頼されていないリファクタを混ぜていないか）
- [ ] 既存機能の実装変更を含んでいないか（含む場合は事前承認が必要）

## 3. 修正前に必ず提示するもの（承認を得てから編集）

1. **原因** — なぜこの修正が必要か
2. **修正箇所** — ファイル名と該当箇所（行番号があれば行番号も）
3. **確認方法** — 修正後にどうやって正しさを確かめるか

あわせて **触るファイル一覧／触らないファイル一覧** を明記する。

## 4. 修正後に必ず報告するもの

1. **変更内容** — 何をどう変えたか
2. **確認方法** — 実際に行った確認と結果
3. **次の確認箇所** — ユーザーが次に見るべき場所

## 5. 回答・報告の形式

- 結論 → 理由 → 数字（変更ファイル数・行数など）の順で書く。
- 推測で断定しない。確認できないことは「確認できません」と書く。

## 6. Git・テストの禁止

- git add / commit / push は行わない（指示があるまで）。`git status --short` 等の確認は可。
- 新規生成テスト（課金が発生する動画・画像生成）は行わない。
- ただし、課金の発生しない動作確認（ビルドが通るか、対象ページがエラーなく表示されるか）は、修正後に必ず行うこと。

## 7. 自動化関連の編集の場合

対象が自動化フロー（自動生成・スケジューラ・Webhook・リトライ処理）に触れる場合は、
編集前に `02_auto_flow_risk_check.md` も必ず通すこと。特に：

- [ ] 重複生成が起きない仕組みがあるか
- [ ] 重複課金が起きない仕組みがあるか
