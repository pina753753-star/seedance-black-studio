# 02. 自動化フロー・リスクチェック

> **読むタイミング：** 自動化フロー（自動生成・自動デプロイ・スケジュール実行・Webhook・自動課金処理）に関わる変更・設計をするときだけ。

自動化フロー（自動生成・自動デプロイ・スケジュール実行・Webhook・自動課金処理）に
関わる設計・変更を行う前に、必ずこのチェックを通すこと。
確認できない項目は「確認できません」と書き、**実行・変更を保留する**。

## 1. 重複防止（最優先）

- [ ] 同じリクエストが2回来ても **重複生成** されないか（冪等キー／リクエストID／DB側の一意制約）
- [ ] リトライ時に **重複課金** が発生しないか（課金処理は成功確認後に1回だけか）
- [ ] ユーザーの二重クリック・二重送信への対策があるか
- [ ] 途中失敗からの再開時に、完了済み処理を再実行しないか

## 2. 暴走・無限ループ防止

- [ ] リトライに上限回数があるか（無限リトライになっていないか）
- [ ] リトライ間隔は適切か（即時連打でAPIを叩き続けないか）
- [ ] 失敗が続いたときの **停止条件** があるか
- [ ] タイムアウトが設定されているか

## 3. 課金・コスト事故防止

- [ ] 1回の自動実行あたりの最大コストを見積もったか（数字で示す）
- [ ] 1日・1時間あたりの実行回数上限があるか
- [ ] 想定外の大量実行を止める手段（キルスイッチ・手動停止）があるか
- [ ] **新規生成テストは行わない**（課金発生のため）
- [ ] ただし、課金の発生しない動作確認（ビルドが通るか、対象ページがエラーなく表示されるか）は、修正後に必ず行う

## 4. データ破壊防止

- [ ] 自動処理が既存データを上書き・削除する経路はないか
- [ ] 削除・上書きがある場合、対象の絞り込み条件は十分に狭いか
- [ ] 失敗時にデータが中途半端な状態で残らないか

## 5. 変更時の手順

- 修正前に **原因・修正箇所・確認方法** を提示し、承認を得る。
- 触るファイル／触らないファイルを明記する（API route・Supabase・OpenRouter関連は指示がない限り触らない）。
- 修正後に **変更内容・確認方法・次の確認箇所** を報告する。
- git add / commit / push は行わない（指示があるまで）。

## 6. 報告形式

- 結論 → 理由 → 数字（リトライ上限・想定コスト・実行回数上限など具体値）の順。
- 推測で断定しない。挙動を確認できない場合は「確認できません」と書く。
