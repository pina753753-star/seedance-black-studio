# 04. エラー報告フォーマット

> **読むタイミング：** エラー・不具合を報告するとき、または修正後に報告をまとめるとき。

エラー・不具合を報告するときは、毎回この型で書くこと。
**事実と推測を必ず区別する。** 確認できないことは「確認できません」と書く。

## 1. 報告の型（この順で書く）

```
■ 結論
（何が起きているか・直せるか直せないかを1〜2行で）

■ 理由
（なぜそう判断したか。根拠となるコード・ログ・エラーメッセージ）

■ 数字
（発生件数・該当ファイル数・行番号・エラーコードなど具体値）

■ 事象
（何をしたら・何が起きたか）

■ 再現手順
（1. 〜 2. 〜 3. 〜。再現できない場合は「再現手順は確認できません」）

■ 確認できた事実
（ログ・コード・git履歴など、実際に確認したものだけ）

■ 確認できないこと・推測
（推測は「推測ですが」を付ける。確認できないものは「確認できません」と明記）

■ 影響範囲
（影響するファイル・機能・ユーザー。重複生成・重複課金の可能性があれば必ず書く）

■ 修正提案（修正する場合）
・原因：
・修正箇所：（ファイル名:行）
・確認方法：
・触るファイル：
・触らないファイル：
```

## 2. 修正を実施した後の報告

```
■ 変更内容
（何をどう変えたか。変更ファイル数・行数）

■ 確認方法
（実際に行った確認と結果。新規生成テストは行わない。ただし、課金の発生しない動作確認——ビルドが通るか、対象ページがエラーなく表示されるか——は必ず行う）

■ 次の確認箇所
（ユーザーが次に見るべき場所・残っている懸念）
```

## 3. 禁止事項

- 推測で断定しない（「〜のはずです」で終わらせず、確認したか否かを書く）
- 確認していないことを「確認済み」と書かない
- git add / commit / push は行わない（指示があるまで）
- 新規生成テストによる再現確認は行わない（課金が発生するため）
- ただし、課金の発生しない動作確認（ビルドが通るか、対象ページがエラーなく表示されるか）は、修正後に必ず行う
- 報告のついでに Flow本体・API route・Supabase・OpenRouter関連コードを修正しない

## 4. 自動化に関わるエラーの場合の追加項目

- [ ] このエラーによって **重複生成** が起きた／起きうるか
- [ ] このエラーによって **重複課金** が起きた／起きうるか
- [ ] リトライが走り続けていないか（現在も実行中でないか）
- 上記が確認できない場合は「確認できません」と書き、先に停止手段を提案する
