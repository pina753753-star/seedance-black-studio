# 03. コード監査チェックリスト

定期監査・リリース前・販売用テンプレート化の前に使うチェックリスト。
**監査は読み取りのみ。** 監査中にコードを修正しない（修正は別タスクとして承認を得てから）。

## 1. セキュリティ

- [ ] APIキー・シークレットがコードに直書きされていないか（OpenRouter / Supabase / その他）
- [ ] `.env` の値がHTML・JS・公開ファイルに漏れていないか
- [ ] 管理者ページ（admin系）が認証なしで到達できないか
- [ ] 販売用テンプレートに個人情報・実キー・実URLが混入していないか

## 2. エラーハンドリング

- [ ] API呼び出しの失敗時処理があるか（try/catch・エラー分岐）
- [ ] 失敗がユーザーに分かる形で表示されるか（無言で失敗しないか）
- [ ] エラー時に課金だけ発生して結果が残らない経路がないか

## 3. 自動化・課金まわり

- [ ] 重複生成を防ぐ仕組みがあるか（冪等キー・一意制約・二重送信対策）
- [ ] 重複課金を防ぐ仕組みがあるか（課金は成功確認後に1回だけか）
- [ ] リトライ上限・停止条件があるか

## 4. 未完成箇所の洗い出し

- [ ] TODO / FIXME / 仮実装 / ハードコードされたテスト値を列挙したか
- [ ] 使われていないファイル・重複ファイルを列挙したか
- [ ] README と実装の食い違いを列挙したか（READMEの修正はまだ行わない）

## 5. 監査報告の形式

- **結論 → 理由 → 数字**（指摘件数・該当ファイル数・行番号）の順で報告する。
- 各指摘に「重大度（高・中・低）」と「該当ファイル:行」を付ける。
- **推測で断定しない。** 実行して確かめていない挙動は「コード上はこう見えるが、動作は確認できません」と書く。
- 確認できないこと（外部サービスの設定・本番環境の状態など）は「確認できません」と明記する。

## 6. 監査中の禁止事項

- コードの修正（修正が必要なら、原因・修正箇所・確認方法を提示して承認を待つ）
- git add / commit / push
- 新規生成テスト（課金が発生するため）
- Flow本体・package.json・API route・Supabase・OpenRouter関連コードへの変更

## 7. 修正フェーズへ進む場合

1. 修正前：原因・修正箇所・確認方法＋触るファイル／触らないファイルを提示
2. 承認を得てから修正
3. 修正後：変更内容・確認方法・次の確認箇所を報告
